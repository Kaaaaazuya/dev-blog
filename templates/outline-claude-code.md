# Claude Code編（D系）の見出し変形例

題材はまずこのブログのリポジトリ自体。パーマリンク引用が可能で、素材収集のためのclone作業が不要。

## 例1: D-1 ブログ運営の全体像（定点観測）

```
## はじめに（「2026年◯月現在」を明記）
## 全体像: Issue → ドラフト生成 → 品質ゲート → AIレビュー → 人間レビュー → デプロイ
   （人間/Claude/CIの担当をレーン分けしたフローチャート）
## 各工程の実際
### write-draftスキルによるドラフト生成
### 品質ゲート（lint / textlint / gitleaks）
### claude-code-actionのAIレビュー
## 設計判断とトレードオフ
   （なぜ最終判断を人間に残すか / 自動化しない工程とその理由）
## まとめ: 次の半年で変えたいこと（→次回の定点観測の予告になる）
```

## 例2: D-2 / D-3 仕組み化の各論

```
## 課題: 同じレビュー指摘を二度受ける / レビュー観点が散らばる
## 全体像: 指摘→スキル反映のループ図（または PR→3並列レビュー図）
## 実装
### スキル・プロンプトの実物（該当箇所を40行以内で引用）
### 実際に回った例（実コミット・実PRを1件たどる）
## 設計判断とトレードオフ
   （non-blockingにした理由 / 観点を分割した理由 / 捨てた案）
## まとめ
```

## 例3: D-6 失敗談集

```
## はじめに: なぜ失敗を公開するか（1段落で短く）
## 試したがやめたこと（1件ごとに: 期待→実際→なぜ捨てたか→今の代替）
### <失敗1>
### <失敗2>
### <失敗3>
## いまも迷っていること
## まとめ: 失敗から残った運用原則
```

## Claude Code編で共通して使える「課題」の書き出しパターン

- 実際に起きた手戻り・レビュー指摘・ヒヤリハットから始める
- 「AIの出力は信用せず、機械チェックと人間レビューで受け止める」前提を明示する
- Before/Afterを工数・指摘件数・手順数など測れる形で示せる記事は必ず示す
